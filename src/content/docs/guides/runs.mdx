A **Run** is an on-demand agent job that fires **once**. You write a prompt, the
coordinator spins up a container and gets to work, and — unlike a
[Schedule](/guides/schedules) — you can **watch its timeline, steer it, and
answer its questions** as it goes.

## Starting a Run

From the orchestrator's **Runs** page, **+ New run** opens the composer:

<Steps>

1. **Instructions** — what you want done (Markdown supported). This is the only
   required field.

2. **Ticket URL** *(optional)* — paste a GitHub/Azure issue and the launch step
   resolves its repo/branch, clones it, and readies the issue context inside the
   container.

3. **Target** *(optional)* — pin a [worker](/concepts/workers) or
   [greenhouse](/guides/greenhouse), **or leave it blank** — the agent will ask
   you to pick a host (and image) the moment it needs to launch.

4. **Engine** — which [engine](/concepts/subagents) runs the task, or
   *auto-detect* to use the best available CLI on the image.

5. **Mode** — **Interactive** (default) or **Autonomous** (see
   [below](#interactive-or-autonomous)).

6. **Start Run** — the coordinator launches a container and begins.

</Steps>

## Interactive or autonomous

A Run can fire in one of two modes:

- **Interactive** *(default)* — a human is watching. The agent can pause and
  [ask you a question](#it-can-pause-and-ask-you), and you can steer it. This is
  everything described on this page.
- **Autonomous** — fire-and-forget. The Run executes to completion **without
  ever pausing** — it's a [Schedule's](/guides/schedules) body fired once, on
  demand. There's no human in the loop, so it never asks a question; if it hits
  an ambiguity it decides for itself or fails fast with a result.

Because an autonomous Run can't ask you anything, it **must be given a target up
front** — a [worker](/concepts/workers) *and* an image, or a
[greenhouse](/guides/greenhouse) (which auto-picks an image). A "just
instructions" autonomous Run with no host is rejected at creation, since it
could only fail when it reached the point of needing to ask.

<Aside type="tip">
  Use **autonomous** for a well-specified one-off you don't want to babysit
  (and where you've pinned the host). Use **interactive** for anything
  exploratory, or where you want to watch, steer, or answer the agent.
</Aside>

## The timeline — watch it work

Every Run has a live **timeline**: each step the agent takes appears as it
happens, so you can follow exactly what it's doing instead of waiting for a
final summary.

<ThemedImage
  dark={timelineDark}
  light={timelineLight}
  title="Runs"
  alt="A completed Run's timeline: assistant messages interleaved with tool steps — launching a container, running the test suite, then opening a PR — with a Run details panel showing the engine (Claude Code — Plan), duration, and summary."
/>

The timeline interleaves the agent's **messages** with its **tool steps** —
each one expandable to see the exact call and result:

- **launch container** — a container is created on the chosen host.
- **exec in container** — a command runs inside it (tests, a build, `gh pr create`).
- **report job result** — the agent finalizes the Run as completed or failed.

The **Run details** panel alongside it shows the status, engine, started/completed
times, duration, the container(s) used, and a one-paragraph summary of everything
the Run did. A finished Run can leave its container up so a follow-up continues
where it left off.

## It can pause and ask you

The headline difference from a Schedule: a Run can **pause mid-flight and ask
you a question**, then resume from exactly where it stopped once you answer. Its
status flips to **Needs you** and the question appears inline.

<ThemedImage
  dark={askDark}
  light={askLight}
  title="Runs"
  alt="A paused Run showing the 'Needs you' status and a question panel: 'Where should the dark-mode toggle live?' with single-select options and a free-text escape — the first of three questions."
/>

How questions are asked:

- **A choice** — single-select (pick one) or multi-select (pick any), each with
  a recommended default and an "Other → free text" escape.
- **Free text** — an open-ended answer.
- **Several at once** — a Run can ask multiple questions in one pause (the
  example above is the first of three).

This works for **any [engine](/concepts/subagents)** you pick — the question
comes from the coordinator, not the engine. The container and its session stay
alive while paused, so it's safe to answer minutes — or hours — later.

<Aside type="note">
  Questions only happen in **Runs**. A [Schedule](/guides/schedules) is
  autonomous, so it never asks — it reports a result and finishes.
</Aside>

## Steering — the chat

A Run is a conversation. Below the timeline is a **chat composer**:

- **Steer it** — type a follow-up instruction while it's working; it's delivered
  at the next turn boundary, so you can redirect without stopping the Run.
- **Answer a question** — when the status is **Needs you**, your reply resumes
  the Run from where it paused.
- **Continue a finished Run** — if the container is still up, a new message picks
  the work back up in the same workspace.

## Picking an engine

Name an engine when you care which one runs. An interactive brainstorm is a good
fit for **Claude Code — Plan** (it can pause itself mid-task); a tightly-scoped
job suits **Claude Code — API credits** or **Codex**. See
[Engines](/concepts/subagents) for the full comparison and how each one pauses.

## Guided example: fix a bug and open a PR

Start to finish, an interactive Run that fixes a failing test:

<Steps>

1. **Open the Runs page** and click **+ New run**.

2. **Write the instruction:**
   > `Reproduce the failing auth test, fix it, and open a PR.`

3. **Paste the ticket URL** (optional) so the container clones the right repo and
   branch.

4. Leave **Target** blank and pick **Claude Code — Plan** as the engine — a good
   fit when you might want a back-and-forth. Keep **Mode: Interactive**.

5. **Start Run.** With no host pinned, the Run pauses almost immediately and
   **asks you where to launch** — choose your worker.

6. **Watch the [timeline](#the-timeline--watch-it-work):** the agent launches a
   container, runs the auth suite (it fails), applies a fix, re-runs (green), and
   opens the PR — each step visible as it happens.

7. If it pauses to **ask a question** (e.g. "scope the fix narrowly or refactor?"),
   answer inline and it resumes.

8. When it reports **completed**, the summary links the PR. The container stays
   up, so you can send a follow-up ("also add a regression test") to continue in
   the same workspace.

</Steps>

Want the same job to run unattended? Pin the worker **and** an image, set
**Mode: Autonomous**, and it runs straight through to a result with no prompts.

## When to use a Run vs a Schedule

- **Run** — exploratory or ambiguous work, anything you want to watch, steer, or
  where the agent may need a decision from you.
- **[Schedule](/guides/schedules)** — recurring, well-specified work that needs
  no supervision.
